Process

How I Started Every Project

Same starting point every time: Screaming Frog crawl + GSC data pull. That combination tells you almost everything wrong with a site in under an hour.

🔄 Standard Start

Crawl the site → filter for 3xx/4xx/5xx, duplicate title tags, missing H1s, images without alt text. Then cross-reference with GSC: find pages that get impressions but have low CTR (usually a meta issue) and pages that rank but have broken schema.

After the crawl, I'd run PageSpeed Insights on the top 5–10 pages by traffic. Most sites had at least 2–3 of the same CWV issues. Then I'd build a simple Google Sheet — issue, impact score (1–5), effort score (1–5), owner. Quick wins go first.

The Google Sheet approach sounds basic but it's what actually got things done — clients could see what was fixed, what was next, and who owned it. No ambiguity.
Frontend Performance

Page Speed & Code Optimisation

This was a big part of the work. Most WordPress sites had render-blocking scripts, uncompressed images, layout-shifting fonts — the usual. I fixed what I could in the theme, and wrote clear dev briefs for the rest.

⚡ LCP Fixes

The most common LCP issue was a 800KB+ hero image loaded without preload. Browser found it too late. Fix was usually two things together:

head-preload.html
<!-- Added to <head> — tells browser to fetch LCP image early -->
<link rel="preload"
      as="image"
      href="/hero-image.webp"
      fetchpriority="high">

<!-- Also converted PNG → WebP. Average size: 820KB → 180KB -->
<img src="hero-image.webp"
     width="1200"
     height="600"
     alt="[descriptive alt text]"
     loading="eager">

The width/height attributes matter for CLS too — without them, browser can't reserve space, so the page shifts when the image loads.

📐 CLS Fixes — Fonts & Layout

Layout shift from Google Fonts was a repeat offender. Fix was simple but had to be done right:

font-loading-fix.html
<!-- Preconnect first — reduces DNS/handshake time -->
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

/* In CSS — prevents invisible text during font load */
@font-face {
  font-display: swap; /* show fallback font, swap when ready */
}

On two sites I also had to fix dynamic JS content being injected above existing page elements — classic CLS cause. Changed the JS to append instead of prepend, which eliminated the shift immediately.

🚀 Render Blocking JS/CSS

WordPress themes load a lot of CSS and JS that blocks rendering. I audited each site's resource waterfall in DevTools and moved what could wait to async/defer.

script-loading.html
<!-- Before — blocking the parser -->
<script src="/js/analytics.js"></script>

<!-- After — doesn't block HTML parsing -->
<script src="/js/analytics.js" defer></script>

<!-- For non-critical CSS — loads async -->
<link rel="preload" href="non-critical.css"
      as="style" onload="this.rel='stylesheet'">

On some WordPress sites, plugins were loading their CSS globally even on pages where they weren't needed. Handled that by conditionally dequeuing scripts on irrelevant pages using WordPress hooks. Nothing fancy — just cleaning up the mess.

🔍 JS Rendering — CSR vs SSR vs SSG

A couple of client sites ran React or Vue frontends. That meant Client-Side Rendering (CSR) — the HTML arrives as a near-empty shell and JavaScript builds the page in the browser. Googlebot can process this, but there's a rendering queue delay that can slow down indexing noticeably.

Where the team had flexibility, I recommended moving toward SSR (Server-Side Rendering) — the server sends fully-built HTML, so crawlers see everything on first request. For mostly-static pages like blogs or landing pages, SSG (Static Site Generation) is the cleanest option: pre-rendered HTML files, nothing to execute, instant crawlability. I used GSC's URL Inspection Tool to check what Googlebot actually rendered after JS execution — usually the fastest way to confirm whether JS content was actually being indexed.

For WordPress-only clients, this wasn't relevant. But for the SPA-style setups, understanding the rendering pipeline was what made the audit actually useful.
Authority & Links

Page Authority & Link Profile

Technical fixes don't move rankings if authority is weak. For a few clients this was the real blocker — not the code.

🔗 Internal Linking & Backlink Audit

Internal linking was the quickest fix. Most sites had decent homepage authority but important service or category pages had almost no internal links pointing to them — link equity just wasn't flowing anywhere useful. I remapped internal linking from pillar pages outward so Googlebot could reach and prioritise the right pages.

On the backlink side, I used Ahrefs to check domain rating, referring domains, and whether inbound links to key pages came from relevant sources. I flagged pages that were technically clean but ranking weakly — the backlink gap was usually why. That went into the audit report as a separate recommendation track.

Canonical tags mattered a lot on eCommerce sites. Filter and sort URL variants were creating dozens of near-duplicate pages and splitting authority across all of them. Canonicalising back to the main URL consolidated the signal properly.

I tracked Page Authority and Domain Rating via Ahrefs and Moz Link Explorer as reference points — useful for explaining to clients why a technically clean page might still be ranking weakly, and why links and content matter alongside the tech.
Structured Data

Schema Injection

Most sites had no JSON-LD at all. I injected it manually — not through plugins, because plugins generate bloated markup that often fails validation.

📄 LocalBusiness — What I Actually Wrote

For local service clients, I wrote the JSON-LD by hand and pasted it into a WordPress Custom HTML block or directly in theme functions.php. Validated every one in Google's Rich Results Test before going live.

local-business.json
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "LocalBusiness",
  "name": "[Client Name]",
  "url": "https://[domain].com",
  "telephone": "+91-XXXXXXXXXX",
  "address": {
    "@type": "PostalAddress",
    "addressLocality": "[City]",
    "addressCountry": "IN"
  },
  "openingHoursSpecification": [{
    "@type": "OpeningHoursSpecification",
    "dayOfWeek": ["Monday", "Friday"],
    "opens": "09:00",
    "closes": "18:00"
  }]
}
</script>

For blogs and finance sites, I used FAQPage schema based on real questions from GSC — the ones getting impressions. That got several pages into the FAQ rich result quickly.
